# BUGFIX_RULES.md — Bug 修复规则

## 核心原则

**修 Bug 的目标是修复问题，不是重写代码。**
修一个 Bug 不允许引入新的 Bug。

## 修复流程

### 1. 定位阶段

- **必须先读完整错误信息**，包括堆栈跟踪、日志、请求/响应内容
- **必须确认 Bug 的复现条件**，不能凭猜测定位
- **必须找到 Bug 的根因**，不能只处理表面症状
- 如果无法确认根因，必须先询问用户或收集更多信息

### 2. 分析阶段

- **必须阅读涉及的完整文件**，不能只看报错的那一行
- **必须理清数据流**：Bug 涉及哪些模块、数据如何传递、在哪里出错
- **必须确认影响范围**：修复这个问题会影响哪些其他功能

### 3. 修复阶段

- **最小改动原则**：只改必须改的代码，一次只解决一个问题
- **不顺手重构**：发现"代码写得不好"不是此时重构的理由
- **不顺手优化**：性能优化、代码美化等与 Bug 修复无关的事不做
- **保持原有逻辑**：除非原有逻辑本身就是 Bug 的根因，否则不改变业务逻辑
- **不删除功能**：不能因为某功能有 Bug 就删除或禁用它

### 4. 验证阶段

- **必须验证修复结果**：确认 Bug 已修复
- **必须验证无副作用**：确认相关功能未受影响
- **必须验证边界情况**：空值、异常输入、并发等场景

## 绝对禁止

| 禁止行为 | 原因 |
|---------|------|
| 重写整个文件来修一行 Bug | 难以审查，容易引入新问题 |
| 同时修多个不相关的 Bug | 无法区分哪个修复导致了问题 |
| 删除报错的代码行 | 掩盖问题，不解决根因 |
| 用 try/except 吞掉所有异常 | 隐藏真正的错误 |
| 改变 API 响应格式 | 破坏前端调用 |
| 修改数据库表结构 | 破坏已有数据 |
| 更换依赖库版本 | 可能引入兼容性问题 |

## 修复记录

每次修复 Bug 后，必须在代码注释或提交信息中记录：
- Bug 现象是什么
- 根因是什么
- 修复方式是什么
- 影响范围是什么
