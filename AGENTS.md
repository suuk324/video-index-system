# AGENTS.md — 项目工作总规则

## 核心原则

每次工作前，**必须先阅读并遵守本文档**，以及本文引用的所有规则文件。

## 必读规则文件

| 文件 | 触发条件 | 说明 |
|------|---------|------|
| [PROJECT_CONTEXT.md](./PROJECT_CONTEXT.md) | **每次工作前必须读** | 了解项目定位、功能边界、技术边界，防止越界开发 |
| [BUGFIX_RULES.md](./BUGFIX_RULES.md) | 遇到 Bug 修复时 | 遵守修复流程，防止修一个坏一个 |
| [CODE_REVIEW.md](./CODE_REVIEW.md) | 每次代码修改后 | 按清单自检，确保修改质量 |
| [PLANS.md](./PLANS.md) | 遇到复杂功能开发时 | 先写计划，确认后再执行 |

## 绝对禁令

1. **不允许无关重构** — 本次任务没有要求重构的代码，一律不改。
2. **不允许大面积乱改** — 一次修改只针对明确的问题点，不做"顺手"改动。
3. **不允许删除功能来假装修复** — 不能因为某功能有 Bug 就删除它，必须正真修复。
4. **不允许凭空猜测项目逻辑** — 遇到不确定的逻辑，必须先读代码或询问用户，不能凭假设行事。
5. **必须优先保护现有功能稳定** — 任何修改不得破坏已有功能。

## 工作流程

### 第一步：理解任务
- 阅读 AGENTS.md（本文）和 PROJECT_CONTEXT.md
- 判断任务复杂度

### 第二步：制定计划（仅复杂任务）
- 如果任务涉及：新增模块、跨文件修改、数据库变更、架构调整
- 按 PLANS.md 模板编写执行计划
- 提交计划给用户确认后再执行

### 第三步：执行修改
- 严格按计划执行（如有计划）
- 每次只改一个明确的问题点
- 不做无关改动

### 第四步：自检
- 按 CODE_REVIEW.md 清单逐项检查
- 确认没有破坏已有功能

### 第五步：Bug 修复（如涉及）
- 严格按 BUGFIX_RULES.md 流程执行

## 验证步骤

- 复杂任务必须包含最终验证步骤
- 验证方式：功能测试、接口测试、页面检查等
- 高风险修改可使用子代理进行独立验证

## 例外情况

以下情况可跳过完整流程：
- 纯文档类修改（不影响代码逻辑）
- 简单的文本修正（错别字、注释修正）
- 用户明确要求快速执行的简单任务

即使跳过流程，**绝对禁令仍然生效**。
